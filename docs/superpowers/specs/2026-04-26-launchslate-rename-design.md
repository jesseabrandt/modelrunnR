# launchslate rename ŌĆö design

**Date:** 2026-04-26
**Status:** Spec only ŌĆö execution deferred until after `feat/stow-unification` merges to main, **and** pending a name decision (see below).
**Name candidates:**
- **`modelrunnR`** (status quo) ŌĆö keep the current name. The maintainer's primary use case is model orchestration, so the name is accurate for that workflow even if the package is technically more general.
- **`launchslate`** (rename) ŌĆö re-frame the package as a general content-addressed run orchestrator, dropping the model framing. Reads as "launch a slate of runs." Available on CRAN (per prior search). Would require the prefix swap below.

If `modelrunnR` is kept, the rest of this spec is a no-op. If `launchslate` wins, everything below applies. CRAN namespace clearance and casing are confirmed at execution time, not now.

## Motivation (for the rename option)

`modelrunnR` is too specific *for users who aren't doing model work*. The package doesn't know or care that it's running models ŌĆö it's a content-addressed run orchestrator usable for any code that produces artifacts. For a hypothetical other user (a data engineer staging ETL runs, say), the name overpromises model-awareness the package doesn't have.

This argument cuts less for the maintainer, whose own primary use *is* model orchestration. The trade-off is real-user discovery (CRAN search, package indexing for non-model audiences) versus name accuracy for the maintainer's own day-to-day use. Currently weighed in favor of the status quo; revisit if the package starts drawing non-model users.

`launchslate` ties to the existing verb (`launch_*`) and reads as "launch a slate of runs." Chosen because it's available (no CRAN clash, no popular-software clash per prior search) and unobtrusive ŌĆö not because "slate" is a load-bearing concept in the API. The prefix `slate_` is needed because bare names like `label()`, `file()`, `hash()` would collide with common base/utility functions.

## Scope

Mechanical rename + prefix swap. **Not** a redesign. Two-tier API preserved:

- **Bare verbs/tables stay bare:** `launch()`, `launch_code()`, `stow()`, `ingest()`, `grab()`, `prune()`, `prune_variants()`, `runs()`, `variants()`, `versions()`, `is_stale()`, `db_path()`.
- **Collision-prone reference/marker constructors get the new prefix:** `mr_*` ŌåÆ `slate_*`.

## Migration policy

**One-release deprecation window.** All `mr_*` user-facing functions remain exported as `.Deprecated()`-warning wrappers that delegate to their `slate_*` replacement. Removed in the release after.

Mechanics:

- Each wrapper looks like `mr_run <- function(...) { .Deprecated("slate_run"); slate_run(...) }`.
- All wrappers live in one file (e.g. `R/zzz-deprecated.R`) with one shared roxygen block; `@keywords internal` so they don't clutter the reference index, `@export` so existing caller code keeps working.
- One consolidated `NEWS.md` entry documenting the rename and deprecation timeline.

## What does NOT get renamed (proposed default)

### DB schema (`_mr_*` tables and result-frame columns)

Tables (`_mr_runs`, `_mr_versions`, `_mr_artifacts`, `_mr_tmp_hash_`) and result-frame columns (`_mr_run_id`, `_mr_variant_label`) **stay as-is**.

**Why:** Renaming the schema requires `ALTER TABLE ŌĆ” RENAME` migrations on every existing `.duckdb` file at connect time. That's real code with real failure modes (partial migrations, concurrent connections, version skew between an updated package and an unmigrated DB) purely in service of internal-name consistency. Tables are underscore-prefixed by convention to mark them as package-internal ŌĆö users don't write them by name. Cost outweighs benefit.

**Trade-off accepted:** Future readers will see `_mr_*` table names and have to know "mr is the old prefix." A short comment in `R/schema.R` documents this.

**If you want schema rename instead:** flag it on spec review. It would become a Phase 2 PR after the rename PR lands, with proper migration logic and a version bump.

## TODO ŌĆö explain option (c) before deciding it's settled

During brainstorming I floated an option (c) that the user marked "idk what c means ŌĆö TODO to explain."

**What (c) was:** today the package has a name collision between `mr_variants()` (the bind constructor ŌĆö produces a list of `mr_variant()` reference objects, used to enumerate variants in a launch call) and `variants()` (the table accessor ŌĆö returns rows from `_mr_versions`). After this rename the collision carries forward as `slate_variants()` vs `variants()`. (c) would have resolved it during the rename ŌĆö e.g. renaming the bind constructor to something like `slate_envelope_variants()`, or renaming the table accessor.

**Why it's deferred:** the collision exists today and hasn't bitten anyone; resolving it requires a separate small design (which name moves, what to call it). Not blocking the rename. Worth a one-paragraph follow-up after the rename lands.
